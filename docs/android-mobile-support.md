# Android Mobile Support

This scaffold keeps Android work isolated from the Python/pygame desktop build until the mobile runtime is ready.

## Gesture Contract

Native Android touch events should be normalized into the same intent stream as desktop input:

- Drag horizontally: steer left/right.
- Drag upward/downward: accelerate/slow.
- Swipe up: jump.
- Swipe down: duck.
- Double tap: push.
- Long press or two-finger tap: request spoken status.

The canonical behavior is implemented and tested in `horse_racing_game/input/touch.py`. Kotlin input code should mirror that contract before feeding commands to the game runtime.

## Shell

- Gradle project root: `android/`
- Application module: `android/app`
- Package/application id: `com.horseracing.audiofirst`
- Debug package suffix: `.debug`
- Release artifact target: Android App Bundle (`.aab`)

The current `MainActivity` hosts the native touch-first race surface with large hit zones, TalkBack labels, haptic hooks, and audio focus handling.

## Race Surface

`RaceSurfaceView` is the first native mobile race surface. It currently:

- Receives Android `MotionEvent` input.
- Mirrors the Python gesture contract into `MobileRaceCommand`.
- Uses `contentDescription` and `announceForAccessibility` for TalkBack.
- Calls `performClick` for tap accessibility.
- Uses haptic feedback when commands are received.
- Draws large full-screen touch lanes instead of small buttons.

The next step is to connect `MobileRaceCommand` to the shared race runtime or a mobile-native runtime bridge.

## Audio And Speech

`AndroidAudioController` is the native mobile audio adapter. It currently:

- Requests and abandons Android audio focus with `AudioFocusRequest`.
- Uses game `AudioAttributes` for low-latency cue playback.
- Uses accessibility speech `AudioAttributes` for `TextToSpeech`.
- Provides `registerCue` and `playCue` around `SoundPool`.
- Speaks status/action labels through Android TTS.
- Handles transient ducking by reducing cue volume.
- Is started/stopped from `MainActivity` lifecycle methods and shut down on destroy.

The next step is to package real Android raw audio resources and map the existing cue catalog into `registerCue` calls.

## Packaging And Release

Android release metadata is generated by `scripts/package_android.py`. Android build-environment readiness is reported by `scripts/check_android_release_env.py`.

Current commands:

- Debug APK: `android\gradlew.bat -p android :app:assembleDebug` on Windows, or `android/gradlew -p android :app:assembleDebug` on Unix.
- Release AAB: `android\gradlew.bat -p android :app:bundleRelease` on Windows, or `android/gradlew -p android :app:bundleRelease` on Unix.
- Install debug APK: `adb install -r android/app/build/outputs/apk/debug/app-debug.apk`
- Launch debug build: `adb shell am start -n com.horseracing.audiofirst.debug/com.horseracing.audiofirst.MainActivity`

`scripts/smoke_android_device.py` automates the connected-device part: it detects `adb`, installs the debug APK, launches `MainActivity`, and returns JSON. Exit code `0` means installed/launched, `2` means skipped because no usable device was connected, and `1` means a real failure.

Release signing uses `android/keystore.properties`, based on `android/keystore.properties.example`. Real keystore files and passwords must stay out of source control.

Device smoke checks:

- Install and launch the debug APK.
- Confirm the race surface is visible and TalkBack can describe it.
- Exercise drag, swipe, double-tap, long-press, and two-finger status gestures.
- Confirm haptic feedback, TTS status output, audio focus handling, and SoundPool cue playback once Android raw cue resources are packaged.
